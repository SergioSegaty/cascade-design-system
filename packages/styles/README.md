# Style

This folder contains the compiled CSS output of the design system's token pipeline. **Nothing in this folder is hand-written** — every file here is generated by the `generator` package from the token source files and should never be edited directly.

## What lives here

```
style/
├── primitives.css      # Raw primitive tokens (color, spacing, font scales)
├── semantic.css         # Semantic tokens (component, layout, stack references)
├── themes/
│   ├── light.css        # [data-theme="light"] overrides
│   └── dark.css          # [data-theme="dark"] overrides
└── index.css            # Aggregated entry point
```

Every generated file starts with:

```css
/**
 * Do not edit directly, this file was auto-generated.
 */
```

If you see this comment and need to change a value, the fix belongs in the token source (in the `generator` package's input files), not here.

## Token model: Primitive → Semantic

The pipeline follows DSCG (Design System Community Guidelines) syntax and works in two layers:

### 1. Primitive tokens
Raw, context-free values — the actual color hex codes, spacing scale, font families. These describe *what the values are*, not what they're for.

```json
"color": {
  "blue": {
    "500": { "value": "#3b82f6" }
  }
}
```

Compiles to:

```css
--color-blue-500: #3b82f6;
```

### 2. Semantic tokens
References to primitives, named by *purpose* or *role*. These describe what a value is *for* — a spacing gap, a component's padding, a theme's text color.

```json
"spacing": {
  "component": {
    "padding": {
      "md": { "value": "{spacing.4}" }
    }
  }
}
```

Compiles to:

```css
--spacing-component-padding-md: var(--spacing-4);
```

Semantic tokens are just aliases — a `var()` pointing back at a primitive. That indirection is the entire point: it's what lets the value underneath change without touching anything that consumes it.

### 3. Theme semantic tokens
A special case of semantic tokens, scoped under a theme selector. They remap a semantic concept (e.g. `text.primary`) to a different primitive depending on active theme.

```json
{
  "color": {
    "text": {
      "primary": { "value": "{color.gray.50}" }
    }
  }
}
```

Compiles to, inside `[data-theme="dark"]`:

```css
--color-text-primary: var(--color-gray-50);
```

Switching `data-theme` on a parent element repoints these variables at runtime — no rebuild, no JS, just the cascade.

## The one rule: components use semantic tokens only

**Components must never consume primitive tokens directly.** Always reach for a semantic variable:

```css
/* ❌ Don't */
.button {
  padding: var(--spacing-4);
  color: var(--color-blue-500);
}

/* ✅ Do */
.button {
  padding: var(--spacing-component-padding-md);
  color: var(--color-text-primary);
}
```

Why this matters: primitives are the implementation detail, semantics are the contract. If a component hardcodes `--color-blue-500`, rebranding or retheming means hunting down every usage. If it uses `--color-text-primary`, updating the token source is enough — the component is automatically update-proof.

The only place primitive tokens should ever be referenced is inside semantic token definitions themselves (the `{spacing.4}` / `{color.gray.50}` references), which is the generator's job, not a component's.

## Regenerating

These files are produced by the `generator` package. To rebuild after a token source change:

```bash
pnpm --filter generator build
```

Never run a manual edit + commit on files in this folder — the next generator run will silently overwrite it, and the diff will be unreviewable noise.

## Adding a new theme

1. Add a new theme semantic JSON file to the generator's token source (following the shape of the dark theme example above).
2. Re-run the generator.
3. A new `[data-theme="your-theme"]` block will appear with its own variables.
4. No component code needs to change, provided components only reference the semantic layer.
